## 第一部分：理论提炼

### 核心思维

1. **代码即沟通**：代码的首要目的是向人解释意图。代码是团队沟通的工具，其次才是让机器执行。

2. **质量是习惯**：代码质量不是最后才考虑的事，而是贯穿整个开发过程。

3. **童子军规则**：离开时让代码比你来时更干净。每次改动都改善一点，日积月累就是大改善。

4. **函数是动词**：函数做事情，类做东西。函数应该小，只做一件事。

5. **错误处理即设计**：如何处理错误是代码设计的重要组成部分。

### 核心观念

1. **命名规范**：
   - 变量名应该揭示意图：为什么存在、做什么用、怎么用
   - 类名应该是名词或名词短语
   - 函数名应该是动词或动词短语
   - 命名是最重要的设计决策

2. **函数设计原则**：
   - 短小：每个函数不超过一屏（20行）
   - 只做一件事：函数应该只做一件事，并且做好
   - 只在一层抽象上操作：不要在函数内混合不同抽象层级
   - Switch语句尽量少用：用多态代替

3. **注释原则**：
   - 好代码>坏代码+好注释
   - 注释是失败的补偿：如果需要写注释解释代码，说明代码本身不清晰
   - 注释会撒谎：代码会改，注释不一定改
   - 真正好的注释：法律注释、解释"为什么"的注释

4. **错误处理**：
   - 不要返回null：返回null会强迫调用者做null检查
   - 不要传递null：null是错误的来源
   - 使用异常而非返回错误码：让错误处理与正常逻辑分离

5. **SOLID原则**：
   - **S**ingle Responsibility（单一职责）：每个类只有一个改变的理由
   - **O**pen-Closed（开闭原则）：对扩展开放，对修改封闭
   - **L**iskov Substitution（里氏替换）：子类可以替换父类
   - **I**nterface Segregation（接口隔离）：不要强迫实现不需要的接口
   - **D**ependency Inversion（依赖倒置）：依赖抽象而非具体

### 关键概念

| 概念 | 定义 | 举例 |
|:---|:---|:---|
| 有意义的命名 | 名称应该表达意图 | isReady vs flag |
| 函数单一职责 | 每个函数只做一件事 | validateAndSave vs validate + save |
| SRP原则 | 单一职责原则 | UserService不应该处理邮件 |
| 开闭原则 | 扩展新功能不需要改现有代码 | 用策略模式替代if-else |
| 童子军规则 | 让代码比你发现时更干净 | 每次改bug顺手改善附近代码 |
| 代码味道 | 暗示可能存在问题的代码模式 | 过长函数、重复代码、过多参数 |

### 底层原理

**为什么命名重要**：
- 代码是给程序员看的
- 好的命名减少认知负荷
- 命名是设计决策的体现

**为什么函数要短小**：
- 短函数更容易测试
- 短函数更易理解
- 短函数降低出错概率

**为什么注释会撒谎**：
- 代码改了，注释不一定改
- 时间长了，注释与代码脱节
- 注释是作者视角，可能误导读者

## 第二部分：实操手册（含推演验证）

### 场景一：命名规范实践

**适用场景**：写代码时给变量、函数、类起名

**执行步骤**：

**步骤1：问自己三个问题**
- 具体操作：给任何事物命名时，问
  1. 为什么要存在？（表达目的）
  2. 做什么用？（表达行为）
  3. 怎么用？（表达用法）
- 如果能回答，说明命名有意义

**步骤2：命名检查清单**
- 具体操作：检查命名是否
  - ✅ 揭示意图而非实现
  - ✅ 发音可读
  - ✅ 无歧义
  - ✅ 无匈牙利命名（type前缀）
  - ✅ 是搜索友好的长度

**步骤3：常见问题修正**
- 具体操作：
  - ❌ getActiveAccount() → ✅ findActiveAccount()
  - ❌ list1 → ✅ accountsWaitingReview
  - ❌ isValid() → ✅ shouldProcessPayment()
  - ❌ UserData → ✅ UserProfile

### 执行前的推演验证

**场景**：命名规范实践

**推演过程**：

1. **如果想不出好名字怎么办？**
   - 推演：说明可能函数/类做了太多事
   - 应对：先拆分，再命名
   - 风险：强行给复杂的东西起名
   - 规避：好命名往往来自好的职责分离

2. **如果团队有不同的命名风格怎么办？**
   - 推演：这是团队协作问题
   - 应对：制定团队命名规范
   - 风险：规范变成教条
   - 规避：重点是"命名有意义"，而非形式统一

3. **边界情况：什么时候用缩写？**
   - 推演：团队通用的缩写可以用
   - 验证：所有人都理解的缩写
   - 结论：业务领域专有名词可以用，社区通用的可以用

**验证结论**：
- ✅ 命名是代码质量的基础
- ⚠️ 关键是揭示意图
- ✅ 命名困难往往是设计问题

### 场景二：函数重构

**适用场景**：发现函数过长或做了太多事

**执行步骤**：

**步骤1：识别"做了不止一件事"的信号**
- 具体操作：
  - 函数名包含"and"
  - 函数超过一屏
  - 注释说明了多个步骤
  - 缩进层级超过3层

**步骤2：拆分函数**
- 具体操作：
  1. 用自然语言描述函数在做什么
  2. 找到"然后"、"此外"、"以及"等连接词
  3. 每一步就是一个小函数
  4. 给每个小函数起描述性的名字

**步骤3：重构示例**
- 具体操作：
  ```python
  # 重构前
  def save_user(user):
      validate(user)
      encrypt_password(user)
      connect_db()
      execute("INSERT...")
      disconnect_db()
      send_welcome_email(user)
      log("User saved")
      
  # 重构后
  def save_user(user):
      validated = validate(user)
      prepared = prepare_for_save(validated)
      persisted = persist(prepared)
      notify_user(persisted)
  ```

### 执行前的推演验证

**场景**：函数重构

**推演过程**：

1. **如果函数拆分后调用层级变深怎么办？**
   - 推演：说明可能过度拆分了
   - 应对：保持在合理深度（<5层）
   - 风险：过度工程化
   - 规避：拆分是为了理解，不是为了形式

2. **如果担心重构引入bug怎么办？**
   - 推演：这是合理的担忧
   - 应对：有测试先跑测试，没有测试先写测试
   - 风险：不敢重构导致代码腐烂
   - 规避：使用IDE重构功能，减少手动错误

3. **边界情况：什么时候不要拆分？**
   - 推演：逻辑简单且稳定的函数
   - 验证：如果拆分后没有增加可读性
   - 结论：拆分是为了可读性，不是为了拆分而拆分

**验证结论**：
- ✅ 函数重构提升可读性和可维护性
- ⚠️ 需要测试保护
- ✅ 关键是"只做一件事"

### 场景三：错误处理设计

**适用场景**：设计错误处理策略

**执行步骤**：

**步骤1：不要返回null**
- 具体操作：
  ```python
  # 不好
  def find_user(id):
      if found:
          return user
      else:
          return null  # 不好！
  
  # 好
  def find_user(id):
      if found:
          return user
      else:
          raise UserNotFoundError(id)  # 好！
  ```

**步骤2：不要传递null**
- 具体操作：
  ```python
  # 不好
  def register(name, email):
      if email is not None:  # 需要检查
          send_email(email)
  
  # 好
  def register(name, email):
      send_email(email)  # email不能为None，要么有要么抛异常
  ```

**步骤3：使用异常而非错误码**
- 具体操作：
  - 正常逻辑和错误处理分离
  - 创建有意义的异常类
  - 异常信息包含上下文

---

### 执行前的推演验证

**场景**：错误处理设计

**推演过程**：

1. **如果调用方真的需要知道"找不到"还是"出错"怎么办？**
   - 推演：用不同类型的异常
   - 应对：
     - UserNotFoundException
     - DatabaseConnectionException
   - 风险：异常类型过多
   - 规避：创建异常层次结构

2. **如果null检查到处都是怎么办？**
   - 推演：说明设计问题
   - 应对：用空对象模式或返回空集合
   - 风险：用null表示"没有"是错误的
   - 规避：明确语义，用合适的方式表示

**验证结论**：
- ✅ null是万恶之源
- ⚠️ 异常处理需要平衡
- ✅ 设计时考虑边界情况

### 场景四：童子军规则实践

**适用场景**：每次修改代码时

**执行步骤**：

**步骤1：改代码前先观察**
- 具体操作：
  - 这段代码想做什么？
  - 有明显的问题吗？
  - 附近的代码有味道吗？

**步骤2：每次改动顺手改善**
- 具体操作：遵循"顺手的"原则
  - 改bug时，顺便改善附近的命名
  - 加功能时，顺便拆分过长的函数
  - 修代码时，顺便删除无用注释

**步骤3：改善但不破坏**
- 具体操作：
  - 每次改善都在可测试范围内
  - 有测试先跑测试
  - 如果改善需要大改，先做计划

---

### 执行前的推演验证

**场景**：童子军规则实践

**推演过程**：

1. **如果改善代码需要很多时间怎么办？**
   - 推演：顺手的改善不应该花太多时间
   - 应对：只做小的改善，大的改善单独安排
   - 风险：过度优化
   - 规避：改善要顺手，不顺手就记录下来

2. **如果改善引入bug怎么办？**
   - 推演：改善应该比原始代码更好
   - 应对：有测试先跑测试
   - 风险：好心办坏事
   - 规避：小步改善，及时验证

**验证结论**：
- ✅ 童子军规则是持续改善的关键
- ⚠️ 关键是"顺手"而非"全面重构"
- ✅ 长期积累效果显著

### 常见错误

| 错误 | 后果 | 正确做法 |
|:---|:---|:---|
| 命名无意义 | 代码难以理解 | 用意图命名 |
| 函数过长 | 难以理解和测试 | 拆分到<20行 |
| 函数做多件事 | 难以复用和维护 | 单一职责 |
| 返回null | 调用方需要大量null检查 | 抛异常或用空对象 |
| 传递null | 到处需要null检查 | 不传null |
| 注释代替好代码 | 注释与代码脱节 | 用代码表达意图 |
| 复制粘贴代码 | 修改时要改多处 | 提取公共函数 |
| 忽视代码味道 | 技术债务积累 | 及时重构 |
